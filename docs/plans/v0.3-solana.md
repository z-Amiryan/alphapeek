## Why this feature (the bet)

Crypto-Twitter cashtag culture is **majority Solana** (pump.fun, `$GOAT`, `$RETARDIO`, `$WIF`).
AlphaPeek is EVM-only today, so a prominent `$GOAT` hover renders **nothing** (the real GOAT is
Solana; our single-match guard correctly stays silent on the EVM knockoffs). For a freshly-shipped
beta, "no data on half the cashtags" is the #1 churn risk. This closes that gap.

**It's already half-built — confirm these before designing (they de-risk the whole plan):**
- **CoinStats serves Solana token data.** Verified live: `GET /v1/coin?coinId=<base58>_solana`
  (e.g. `$TRUMP`) returns a full token card (price + 7d chart). So the token-data half is FREE and
  working — we just can't *detect* Solana or *safety-scan* it yet.
- **DexScreener covers Solana** (`chainId: "solana"`) — the coverage fallback already gets Solana
  pairs; it's gated out only because `DEX_CHAIN` (apps/worker/src/dexscreener.ts) lists EVM chains.
- **CoinStats `/coins?symbol=` returns Solana coins** with `contractAddresses:[{blockchain:"solana",…}]`.

## Scope

**IN (this cut — v0.3.0):**
1. **Solana token cashtags** — `$GOAT` etc. resolve to the token card. (Whitelisted Solana coins
   already work after the v0.2 `4b35789` coinId-case fix; the long-tail path needs widening — below.)
2. **Solana token-address (mint) detection** — hover a pasted base58 mint → token card.
3. **Solana contract safety** — keep the safety-first VP on Solana (RugCheck or GoPlus-Solana).
4. **DexScreener fallback for Solana** mints (fresh/unindexed).

**OUT (defer to a follow-up cut — note, don't build):**
- **Solana wallet balances / PnL.** Needs a separate CoinStats wallet path (verify via
  `get-blockchains` for a Solana `connectionId`) or a Solana RPC. Cashtag culture is token-centric;
  wallets are lower-value and higher-effort. Detect a base58 *wallet* → show a minimal "wallet"
  affordance or just skip in v0.3.0.
- Non-EVM, non-Solana chains (Tron, etc.). Different cut.
- Inline badge mode, watchlist — separate roadmap items.

## The one real risk: base58 detection false positives

EVM detection is trivial (`0x` + 40 hex). Solana mints are **32–44 base58 chars with no prefix**, so
a naive regex will match other base58 strings (other chains' addresses, IPFS CIDs, random tokens).
**Do NOT show a card on raw detection.** Reuse the v0.2 **pre-flight pattern** (see `content.ts`
`showCard` for the long-tail `$symbol` path): detect a *candidate*, ask the Worker if it resolves to
a real Solana token, and **mount only on a confirmed token** — with the **underline deferred** until
confirmation. False positives then cost a cached Worker probe, never a wrong card or a flashed cue.
This is the same trust mechanism already shipped for long-tail cashtags; lean on it.

Detection regex (starting point — verify/tune): `(?<![1-9A-HJ-NP-Za-km-z])[1-9A-HJ-NP-Za-km-z]{32,44}(?![1-9A-HJ-NP-Za-km-z])`.
A strong high-confidence signal: pump.fun mints end in `pump`. Consider requiring either the `pump`
suffix OR a successful pre-flight before showing — measure false-positive rate on real tweets.

## Phases (commit after each, with self-review + live self-test)

### Phase 0 — Lazy-load the hover card (prerequisite)
The content script is ~83 KB gz vs the SPEC §7 25 KB budget because React + the card are eager in
the content entry. Solana adds a detection path (and likely a longer whitelist), making it worse.
**Fix first:** dynamic-`import()` the card module (`mount.ts` → `HoverCard` + its deps) on first
hover, so the content entry drops to ~20 KB. WXT/Vite support dynamic import in content scripts;
mind MV3 `web_accessible_resources` for the lazily-loaded chunk. **This touches the crown-jewel
hover loop — keep the existing address/cashtag behavior byte-identical; the show-generation guard
and scroll-dismiss must still work.** Verify the bundle with `pnpm --filter @alphapeek/extension build`
and confirm content.js < 25 KB gz. *(If this proves too invasive, ship Solana without it and track
the budget as a known overage — but try first.)*

### Phase 1 — Shared types + chain plumbing
- `packages/shared/src/types.ts`: introduce a Solana notion. Cleanest: add `'solana'` to a broader
  union without polluting the EVM `Chain` (which drives EVM-only slug maps). Options: a separate
  `type SolChain = 'solana'` or a `Network = Chain | 'solana'`. Pick the one that keeps `SUPPORTED_CHAINS`
  (EVM) intact — many maps (`COINS_CHAIN`, `WALLET_CHAIN`, GoPlus ids, `DEX_CHAIN`) are EVM-keyed.
- Decide how a Solana token card carries its chain for links (solscan.io, not etherscan).
- `apps/extension/src/lib/chain.ts`: add `solscanTokenUrl(mint)` and a DexScreener-Solana link.

### Phase 2 — Worker: Solana token resolution + safety
- **DexScreener:** add `solana` to `DEX_CHAIN` (apps/worker/src/dexscreener.ts) so the existing
  fallback returns Solana tokens. The pair `chainId` is already authoritative.
- **Long-tail symbol guard (`pickSymbolMatch` in coinstats.ts):** today it requires a *supported EVM*
  deployment. Widen it to also accept a `solana` deployment (so `$GOAT`-class long-tail cashtags can
  resolve to the dominant Solana coin under the same single-match + market-cap-floor discipline).
  **Keep the precision guard** — silent on contested tickers.
- **New safety provider for Solana.** Two free options — verify both, pick the better-calibrated:
  - **GoPlus Solana** (`/api/v1/solana/token_security?contract_addresses=…`) — reuse the goplus.ts
    pattern + one provider. Verify it returns useful Solana fields.
  - **RugCheck** (`https://api.rugcheck.xyz/v1/tokens/{mint}/report`) — Solana-native, free, detailed
    (risks, LP lock, top-holder concentration). Map its output to the existing `TokenSafety` shape.
  Calibrate against a trusted Solana basket (e.g. WIF, BONK, JUP, POPCAT) the same way GoPlus was
  calibrated for EVM — informational notes must not drive the verdict. Extend `TokenSafety.source`
  to include the chosen provider; keep best-effort (card renders without it).
- **A Solana token-resolution route.** Either extend `resolve()`/`buildToken` to accept a Solana mint
  (token-first, like the EVM path) or add `GET /v1/sol?mint=…`. Validate the mint with a base58
  regex (server-side), `encodeURIComponent`, under the existing `/v1/*` rate-limit + daily-cap
  middleware. Negative-cache non-resolving mints (mirror the `symid:` sentinel) so detection false
  positives don't burn the request cap.

### Phase 3 — Extension: detection + pre-flight wiring
- `lib/regex.ts`: `findSolanaMint(text)` (base58, the regex above). `content.ts` `detectTarget`:
  after the `0x` and `$cashtag` branches, add a `solana-mint` candidate branch. Add a `Target` kind
  `{ kind: 'sol'; mint: string }` (mirror the existing union in `shadow/mount.ts`).
- **Pre-flight + deferred underline** for the `sol` kind in `showCard` — copy the `symbol`-kind
  branch exactly (probe the Worker, mount only on a `token` result, add the underline after
  confirmation). This is what makes base58 detection safe.
- `messaging.ts`/`worker-client.ts`/`background.ts`/`cache.ts`: add the Solana lookup message +
  worker call + an IndexedDB cache key (`sol:{mint}`), mirroring the `SYMBOL_LOOKUP`/`sym:` trio.
- `TokenView`: Solana cards link to **solscan** (no EVM explorer); reuse the `source`-based footer.

### Phase 4 — Docs + store
- SPEC §4/§5 (new route, `DEX_CHAIN` += solana, safety provider, base58 detection + pre-flight),
  §8 smoke rows, §9 (Solana gap → closed for tokens; wallets still deferred). ROADMAP (move Solana
  token support into the built cut; note wallets deferred). privacy.html/md (disclose the new safety
  provider — RugCheck/GoPlus-Solana — as a third-party recipient of the hovered mint, **mandatory**).
  README + store-listing (Solana now supported). Refresh `apps/store-visuals` with a Solana example
  if useful.
